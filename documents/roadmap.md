# mini-atoms P0 Roadmap（今日冲刺版）

版本：v1.0
日期：2026-02-24
目标：今天完成可在线演示的 P0 闭环（自然语言 -> 草稿生成 -> 多轮对话 -> 发布 -> 分享）

---

# 1. 今日目标（Definition of Done）

今天结束时，系统应满足以下条件：

- 用户可注册/登录（本地账号密码）
- 可创建项目并输入自然语言需求
- LLM 能生成合法 Spec 并保存为草稿
- 同一项目可继续对话并重新生成草稿
- 前端可展示 AI 正在工作的临时动画/逐字效果（不落库）
- 基于 Spec 渲染可运行应用（至少支持 List / Form / Toggle / Stats）
- CRUD 可用且数据持久化（SQLite）
- 可发布到 `/p/{slug}`
- 可分享到 `/share/{slug}`，且分享页只读
- 完成 build / test / lint / 基本 e2e 验证（含截图）
- 在线 Demo 可访问并可演示完整闭环

# 3. 执行策略（今天能做完的关键）

- 按模块串行推进，避免大面积并行导致返工
- 每个模块先确认方案，再编码，再跑最小验收
- 关键模块坚持 TDD（至少先写核心单测）
- 优先闭环能力，UI 精修放到最后
- `Timer` 采用简化版（前端计时 + 手动写入记录）
- AI 动画只做前端效果，不设计后端事件存储

# 4. 模块拆分与顺序（主线）

## M0. 项目骨架与工程基线（优先级 P0）

目标：

- 搭好 Go 服务骨架、配置、模板、路由、SQLite 迁移机制
- 建立可重复执行的 `Makefile`

交付物：

- 项目目录结构（`cmd/`, `internal/`, `web/templates`, `migrations`）
- 配置加载（环境变量）
- 基础路由与健康检查
- SQLite 初始化/迁移执行入口
- `Makefile`：`build` / `test` / `lint` / `dev`

方案确认点（你确认）：

- 目录结构是否接受
- 路由组织方式（按模块还是按页面）
- 配置项命名（`DATABASE_PATH`, `SESSION_SECRET`, `DEEPSEEK_API_KEY` 等）

验收标准：

- 服务可启动
- SQLite 文件可创建
- 迁移可重复执行（幂等）
- `make build` 可通过（至少当前骨架）

预计耗时：

- 45-60 分钟

目标：

交付物：

- `users` / `user_sessions` 表迁移
- 注册页、登录页、退出接口
- bcrypt 密码哈希

测试（先写）：

- 密码哈希/校验单测
- Session 创建/过期校验单测
- 注册/登录 handler 基本用例测试

方案确认点（你确认）：

- Session 有效期（例如 7 天）是否确认

验收标准：

- 可注册新账号
- 可登录并进入受保护页面
- 退出后受保护页面需重新登录

预计耗时：

- 60-90 分钟

## M3. Spec Schema、Validator 与 Collection 同步策略

目标：

- 固化最小 Spec 能力，确保后续渲染与 CRUD 可控

交付物：

- Spec 结构体定义（Go）
- Validator（原语白名单、字段类型、数量限制、block 引用校验）
- `collections` / `records` 表迁移
- Collection 同步逻辑（仅兼容变更）

测试（先写）：

- 合法 Spec 通过
- 非法 primitive 拒绝
- `toggle` 引用非 `bool` 字段拒绝
- `stats sum` 引用非数值字段拒绝
- 不兼容 schema 变更拒绝（至少 1-2 个用例）

方案确认点（你确认）：

- P0 默认拒绝的不兼容变更清单（删字段/改类型）
- 错误提示文案风格（偏技术/偏用户友好）

验收标准：

- 给定示例 Spec 能通过校验
- 非法 Spec 明确报错
- Collection 同步不会破坏已有数据结构

预计耗时：

- 90-120 分钟

## M4. LLM 生成流程 + Chat 持久化（核心）

目标：

- 打通“输入需求 -> 调用 LLM -> 校验/修复 -> 保存草稿 -> 写入聊天记录”

交付物：

- `chat_messages` 表迁移与 repo
- 生成服务（构造 prompt、调用 LLM、解析 JSON、校验、一次修复重试）
- 写入 `chat_messages`：
  - `user`：用户输入
  - `assistant`：面向用户的 AI 回复摘要（可由服务端生成）
  - `system`：失败错误（如有）
- 保存 `projects.draft_spec_json`

测试（先写）：

- LLM 客户端接口 mock 测试（成功 / 非法 JSON / 校验失败 / 修复成功）
- 聊天记录写入顺序测试
- 失败时草稿不覆盖旧值测试

方案确认点（你确认）：

- `assistant` 回复展示风格（简短摘要 / 包含变更说明）
- 修复重试次数（P0 固定 1 次）

验收标准：

- 输入自然语言需求后能生成草稿 Spec
- 聊天区看到用户消息和 AI 回复
- 失败时展示错误且不破坏旧草稿

预计耗时：

- 90-120 分钟

## M5. Primitive Renderer + CRUD（List/Form/Toggle/Stats）

目标：

- 完成 P0 可运行应用能力，形成可用闭环

交付物：

- 统一渲染入口 `RenderApp(spec, mode)`
- Primitive 渲染器：
  - `nav`
  - `list`
  - `form`
  - `toggle`
  - `stats`
- CRUD 接口（创建记录、列表、更新布尔字段等）
- `share_readonly` 模式下写操作拦截

测试（先写）：

- CRUD 字段类型校验单测
- `toggle` 更新接口单测
- `stats count/sum` 计算单测
- Share 只读写接口返回 403 测试

方案确认点（你确认）：

- 列表/表单最简 UI 结构
- Stats 展示样式（数字卡片即可）

验收标准：

- Todo/读书记录类应用可新增、列表查看、切换布尔字段
- Stats 可显示 `count`（以及 `sum` 若 Spec 需要）
- Share 页面无法写入

预计耗时：

- 120-180 分钟（今天最长模块）

## M6. 多轮对话 UI 与 AI 动画（前端临时效果）

目标：

- 提升生成体验，满足“AI 正在干活”的感知需求

交付物：

- 聊天输入框与消息列表 UI
- 提交后 loading / typing 动画
- 处理中轮播文案（例如“理解需求 / 生成结构 / 校验渲染”）
- 完成后刷新聊天区与预览区

说明：

- 不落库存储动画状态

测试：

- Playwright 覆盖 1 条生成流程（看到 loading/typing 元素出现并消失）

方案确认点（你确认）：

- 动画风格（dots / typing / skeleton）
- 文案风格（偏专业/偏亲和）

验收标准：

- 提交需求后能看到明显忙碌态
- 完成后动画收起且结果刷新
- 失败时有清晰提示

预计耗时：

- 45-75 分钟

## M7. 发布与分享（双态 + 只读）

目标：

- 完成 `Publish` 与 `Share` 闭环，并确保分享页后端只读

交付物：

- 发布动作（复制 `draft_spec_json -> published_spec_json`）
- `published_slug` / `share_slug` 生成逻辑
- 路由：
  - `/p/{slug}`
  - `/share/{slug}`
- 分享页内容：
  - 聊天记录（用户 + AI + system）
  - 当前草稿预览（只读）
  - 当前已发布预览/状态（若存在）
  - 可选 Spec JSON 展示
- 服务端只读保护（`share_readonly`）

测试（先写）：

- 发布逻辑单测（首次发布 / 再次发布覆盖）
- Share slug 生成与复用单测
- `/share/*` 写操作 403 测试

方案确认点（你确认）：

- Share 页面内容布局与是否默认展示 Spec JSON
- 发布后 `/p/{slug}` 页面是否允许写入（当前方案：允许）

验收标准：

- 点击 Publish 后 `/p/{slug}` 可访问
- 点击 Share 后 `/share/{slug}` 可访问且无需登录
- 分享页写操作被后端拒绝

预计耗时：

- 60-90 分钟

# 7. 风险与应对（今日冲刺版）

风险 1：LLM 输出不稳定导致生成失败

- 应对：强约束 JSON + Validator + 一次修复重试 + 保留旧草稿

风险 2：Renderer + CRUD 开发超时

- 应对：先锁定 `List/Form/Toggle/Stats` 最小功能，不做复杂样式和过滤排序

风险 3：分享页只读遗漏后端校验

- 应对：写接口统一加模式检查，Playwright 增加只读验证

风险 4：Timer 耗时超出预期

- 应对：坚持简化实现；如时间极限，先保证主闭环（M0-M7、M9）完成，再补 M8
